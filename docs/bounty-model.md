# The non-money AI bounty board

**Status:** proposal · **Owner:** founder · **Last updated:** 2026-07-13
**전신:** poolproof (pay-on-green escrow) → 판별 게임 → **다시 poolproof.**
바운티 = poolproof 원래 구조 그 자체. "escrow가 green에 풀린다" =
"AI가 만들어서 테스트 통과하면 바운티 지급." 판별게임을 걷어내고 코어로 복귀,
단 **현금을 뺀다.**

---

## 1. 한 줄 정의

> **"AI로 이거 만들어줘"를 바운티로 올린다 → 누구의 AI든 실제로 만들어
> 숨은 테스트(holdout)까지 통과하면 바운티 획득. 현금이 아니라 크레딧으로.**

핵심은 **"AI가 진짜 뭘 만들 수 있냐"** 를 실행으로 정산하는 것. 데모·말이
아니라, 실제 코드가 실제 테스트를 통과해야 바운티가 나간다.

---

## 2. 미국에 이미 있다 — 그리고 우리가 다른 점

| | Algora | TaskBounty | **poolproof** |
|---|---|---|---|
| 바운티 | GitHub 이슈 `/bounty $1000` | AI가 자동 해결 | "AI로 X 만들기" |
| 판돈 | **현금 (USD)** | 현금 | **크레딧 (현금화 불가)** |
| 검증 | **사람이 PR 머지 승인** | 자동 | **holdout 자동 판정** |
| 규제 | 결제·세금·KYC | 결제 | **없음 (돈 안 오감)** |

Algora는 $250k+ 지급하며 이 시장이 진짜임을 증명했다. 우리의 두 차별점:

1. **돈이 아님.** 판돈 = 현금화 안 되는 크레딧. 결제 레일·규제·도박선 전부
   회피 (legal 문서의 "no payouts"와 일치). 그래도 베팅의 손맛은 진짜 —
   크레딧은 한정돼 있고, 잔고가 곧 실력·평판.
2. **사람 승인이 아니라 holdout 자동 검증.** Algora는 메인테이너가 주관적으로
   머지해줘야 한다(느리고, 봐주기 가능). 우리는 **공개 안 한 숨은 테스트**가
   객관적으로 "진짜 만들었나"를 판정. 봐주기도, 게이밍도, 오염도 불가.
   `specs/josa`가 실례: public(책→책은)은 통과해도 holdout(서울+(으)로→서울로)에서
   죽으면 바운티 안 나감 = "되는 척"은 못 탄다.

---

## 3. 돈 아닌 판돈 — earn-to-post 크레딧 경제

현금이 없으면 "왜 남의 바운티를 풀어주나?"가 문제. 답은 **닫힌 크레딧 순환**:

```
바운티 푼다 (AI가 통과) ──► 크레딧 획득 ──► 그 크레딧으로 내 바운티를 건다
        ▲                                              │
        └──────────  남이 내 바운티를 풀어준다  ◄────────┘
```

- **크레딧을 벌려면** 남의 바운티를 (자기 AI로) 풀어야 한다.
- **내 문제를 AI가 풀어주길 원하면** 먼저 크레딧을 벌거나, 걸어야 한다.
- 현금 0으로 2-sided 마켓이 부트스트랩. 잔고·해결 전적 = 공개 실력 증명
  (Kaggle 랭킹·StackOverflow rep의 바운티판). 이게 "돈 안 걸고 뭔가 건다".

시드: 신규 가입 시 소액 크레딧 지급 → 첫 바운티를 걸 수 있게. 인플레 방지는
"바운티 지급액 = 출제자가 건 액수"로 zero-sum 유지.

---

## 4. 참여 3역할

- **출제자(poster).** "AI가 이거 만들 수 있을까?" 를 스펙+holdout으로 올리고
  크레딧을 건다. 어려운(=holdout이 매운) 바운티일수록 화제.
- **빌더(solver).** 자기 에이전트/모델을 붙여 제출 → 자동 실행 → holdout까지
  통과하면 바운티 획득. green도 실패도 이름 붙어 공개 전적.
- **구경꾼→참여자.** "내 AI도 저건 풀겠는데" → 빌더 전환. (Algora의 UGC 루프.)

## 5. 무엇을 버리나

- 사람 vs AI **판별 게임** 전체 (`/play`, streak, 🟩🟥 공유 그리드, 랭크).
- **현금/결제** 레이어 (`polar.ts` escrow 지급) — 크레딧으로 대체.

## 6. 이미 있는 것 vs 만들 것

**있는 것 (거의 그대로 재사용):**
- `runner.ts` + `_harness.mjs` + `specs/*/{public,holdout}.test.mjs` — 실제 실행
  + holdout 자동 판정. **바운티 검증 엔진이 이미 완성.**
- `projects`(=바운티), `pledges`(=건 크레딧), `verification_runs`, `slots`,
  `ledger` — 데이터 모델이 이미 바운티 구조.
- `grid.ts` — 실행 결과 압축.

**만들 것 (build order):**
1. **랜딩·용어 리프레임** — 현금 escrow → 크레딧 바운티. (이 PR)
2. **크레딧 원장** — 가입 시드, 바운티 걸기(hold), 통과 시 solver에게 지급.
   `ledger`/`pledges` 위에 현금 제거 + 크레딧 규칙.
3. **바운티 상세·제출 흐름** — 바운티 걸기(출제) / 풀기(제출→실행→정산).
4. **라이브 모델 실행** — `ai.ts`로 실제 모델을 붙여 자동 제출(옵션).

## 7. 북극성

**정산된 바운티 수** (AI가 실제로 만들어 holdout까지 통과한 건). 부차:
크레딧 순환 속도(벌고-거는 회전), holdout이 매워 여러 AI가 실패한 "진짜 어려운"
바운티 수. 판별게임 지표(Daily Plays·공유율)는 폐기.
